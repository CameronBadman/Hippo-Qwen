Hippo-Qwen Next Ideas
=====================

Goal
----

Make Hippo-Qwen stronger as a deterministic agent memory system for Desygner,
Latitude 37, and the Qwen MemoryAgent track.

The current benchmark story is much cleaner after the leakage fixes and
held-out-family retest. The next bottleneck is not only evidence recall. Ranking
is already strong in the small MemoryCraft runs. The larger issue is context
quality: the agent-facing context still contains too much noise.


Immediate Measurement Work
--------------------------

1. Failure attribution metric

Add evidence-in-pool rate to every benchmark.

Purpose:
- Separate retrieval/index failures from calibrator failures.
- If evidence never reaches the candidate pool, the reranker cannot fix it.
- At larger memory counts, this will likely become the real bottleneck.

Metrics:
- evidence_in_pool@candidate_pool
- evidence_missing_from_pool_count
- evidence_ranked_out_count
- evidence_retrieved_count

Measure evidence-in-pool at two points so the failures do not get conflated:
- at the raw union retriever output
- after truncation to calibrator_max_candidates

Interpretation:
- Low evidence-in-pool at union output means improve the index/candidate
  generator.
- Evidence lost between union output and the truncated pool means raise or
  rework the truncation, not the calibrator.
- High evidence-in-pool but low recall@k means improve the calibrator.


2. Set-attention ablation

Train and evaluate a pointwise MLP calibrator against the current set
transformer.

Purpose:
- Prove whether joint candidate attention is actually adding value.
- If set attention wins on near duplicates, superseded conflicts, and wrong-time
  memories, that becomes the technical novelty claim.
- If it does not win, shrink the model and reduce latency.

Experiment:
- Same inputs and training data.
- Model A: current 3-layer set transformer.
- Model B: num_layers=0 or pointwise MLP per candidate.
- Compare per adversarial profile.

Profiles to watch:
- near_duplicate
- superseded_conflict
- same_entity_wrong_time
- stale_preference


3. Abstention benchmark

Stop skipping abstention rows by default in at least one benchmark path.

Purpose:
- Agent memory systems must know when no relevant memory exists.
- False memories are worse than missed memories in many product settings.

Metrics:
- false_memory_rate
- abstention_precision
- abstention_recall
- no_relevant_context_rate
- average returned memories for abstention queries

Product framing:
- "Hippo-Qwen can deterministically say there is no safe memory to use."


4. Credibility prerequisites carried over from the held-out retest

These are not new ideas, but every externally shown number should eventually
come from this configuration. Do not let them drop out of Phase 1:

- Hippo encoder instead of hash embeddings.
- Larger holdout with more questions per scenario.
- Off-the-shelf cross-encoder reranker baseline (for example bge-reranker)
  in every comparison table, so the claim is "beats the standard retrieval
  stack", not "beats raw ANN".
- Mutated real-turn hard negatives (covered by idea 7) as the eval stressor.


5. Paraphrase consistency

Measure whether the same memory is retrieved when a user asks the same question
in different words.

Purpose:
- Determinism covers identical queries.
- Agents often rephrase questions.
- A useful memory system should be stable under semantic paraphrase.

Metrics:
- top-k overlap across paraphrases
- evidence recall variance across paraphrases
- context-frame overlap

Possible training change:
- Add paraphrase-augmented query rows if consistency is weak.


Highest-Value Build Work
------------------------

6. Budget packing and learned stop

Attack context precision directly.

Current issue:
- Recall@8 and MRR are strong.
- Context precision is much weaker.
- The returned agent context can still be 60-75% noise.

Build:
- Train the include/utility head on marginal value per token.
- Add a learned stop threshold.
- Length-penalize bloated candidates.
- Return 3 memories when 3 are enough instead of always filling top-k.

Metrics:
- context_precision
- context_recall
- included_count
- tokens_returned
- evidence_tokens_per_context_token
- false_memory_rate

Product framing:
- "Smaller, cleaner context for agents."


7. Mined hard negatives

Replace most synthetic training templates with real near-misses.

Sources:
- The union retriever's top-ranked non-evidence candidates.
- Evidence for other questions in the same record.
- Near misses from FAISS/token union.
- Same-session non-evidence turns that share query terms.

Purpose:
- Remove training template leakage risk.
- Train on realistic failure cases.
- Keep templates only as held-out eval stressors.

Expected gain:
- Better transfer to real memory workloads.
- More credible benchmark methodology.


8. Temporal encodings

Give the set transformer a better representation of time and order.

Current issue:
- Time mostly enters as scalar age_days.
- Stale-preference and wrong-time failures are relative.
- The model needs to learn "newer memory on same topic supersedes older memory."

Build:
- Add timestamp/session-order encodings per candidate.
- Try sinusoidal or learned temporal embeddings.
- Include relative order features within the candidate set.

Profiles to watch:
- stale_preference
- same_entity_wrong_time
- superseded_conflict


9. Cross-candidate explicit features

Add deterministic features that make set relationships easier to learn and
easier to explain.

Features:
- max cosine similarity to a higher-ranked candidate
- near-duplicate count in the candidate set
- shares entities with newer higher-scored candidate
- shares entities with older lower-scored candidate
- same session as top candidate
- same speaker as top candidate
- same entity but different timestamp bucket

Purpose:
- Small model reliability.
- Better trace explanations.
- Less pressure on attention to infer everything implicitly.


10. Supersedes edges at write time

Make corrections load-bearing instead of purely learned.

Build:
- When feedback marks a memory corrected, create an explicit supersedes edge.
- Expose is_superseded and superseded_by_newer_candidate as hard features.
- Use the edge during retrieval and context packing.

Purpose:
- Convert a major failure mode into deterministic behavior.
- Improve explainability.


Novel Research Directions
-------------------------

11. Counterfactual replay for memory credit assignment

This is probably the strongest novel direction.

Core idea:
- Because Hippo-Qwen is deterministic, replay a past query with one memory
  removed and measure whether the downstream outcome changes.
- If the outcome changes, that memory had causal utility.
- If the outcome does not change, that memory was probably noise.

The outcome variable must be downstream, not retrieval composition. Removing
a returned memory always changes the returned context, so "did the context
change" is circular and would mostly relabel the existing ranking. Utility is:

  utility(m) = outcome(context with m) - outcome(context without m)

where outcome is answer-level: does Qwen (temperature 0) still produce the
gold MemoryCraft answer, or at minimum does the remaining context still
contain sufficient evidence for it. Determinism makes the replay
reproducible; the gold answers make it gradeable.

Redundancy trap (leave-one-out is not enough):
- Two near-duplicate memories that each cover the same fact both get zero
  leave-one-out utility, because the other one covers the removal.
- Naive labels would mark both as useless, the budget packer would drop
  both, and the answer would break.
- Fix: dedup or group correlated memories first and ablate groups, or
  sample small coalitions Shapley-style.

Why it matters:
- Creates causal utility labels instead of heuristics.
- Feeds budget packing.
- Feeds safe forgetting.
- Gives a research-shaped claim: counterfactual credit assignment for
  deterministic agent memory.

First experiment:
- On MemoryCraft holdout, retrieve context for each query.
- Group near-duplicate memories, then remove one group at a time.
- Replay the query through Qwen at temperature 0.
- Label each group by whether the gold answer survives.
- Train the utility/include head on those labels.
- Measure context precision, tokens returned, and answer survival rate.

Cost note: this is answer-level replay, so it needs LLM calls per ablation.
It is an order of magnitude more work than ideas 1-2. Budget for it as a
phase of its own rather than a quick experiment.


12. Provably safe forgetting

Define forgetting as compression under a replay constraint.

Core idea:
- A memory may be deleted only if counterfactual replay shows its removal
  changes no past decision beyond a threshold.

Product claim:
- "We forget aggressively, and can show why each deletion was safe."

Why deterministic memory matters:
- Non-deterministic retrieval stacks cannot cleanly prove this.


13. Versioned, branchable memory

Expose git-like memory state semantics.

Features:
- memory-state hashes
- branch
- merge
- diff
- replay under old memory state
- pin output to memory-state hash

Use cases:
- Audit why an agent changed behavior.
- Test "what would the agent do if it believed X?"
- Reproduce a design or compliance decision.


14. Learned consolidation at write time

Move intelligence from only retrieval time into the memory write path.

Core idea:
- A second set transformer runs over a new memory plus its retrieved
  neighborhood.
- It emits deterministic mutation decisions:
  - merge
  - supersede
  - keep both
  - discard
  - summarize into semantic memory

Why it matters:
- Timely forgetting.
- Smaller memory store.
- Better retrieval because bad memories are handled before query time.

Experiment:
- Build supervision from MemoryCraft session structure and correction-like
  patterns.
- Compare retrieval quality on raw append-only store vs consolidated store.


15. Relational output heads

Return a structured memory frame instead of only a ranked list.

Build:
- Add pairwise relation heads over the candidate set.
- Predict:
  - A supersedes B
  - A duplicates B
  - A contradicts B
  - A supports B
  - A is same-topic older context for B

Agent-facing output:
- facts
- preferences
- corrections
- replaced memories
- contradictions
- evidence confidence

Why it matters:
- Better than handing the agent undifferentiated snippets.
- Makes trace output self-explanatory.


16. Compressed memory tokens

The tractable version of the LoRA memory idea.

Core idea:
- Compress the selected memory subgraph/frame into a handful of soft tokens or
  KV-prefix entries.
- The agent consumes memory tokens instead of hundreds of raw text tokens.

First experiment:
- Distill Qwen answers with full text context into answers with compressed
  memory tokens.
- Measure answer equivalence and token savings.

Risk:
- More research-heavy.
- Should come after cleaner retrieval/context packing.


Additions After Review
----------------------

17. End-to-end answer accuracy benchmark

Retrieval metrics are a proxy. The Qwen track judges agent outcomes, Desygner
will judge agent outcomes, and counterfactual replay (idea 11) needs the same
machinery anyway.

Build:
- Pipe the returned context into Qwen at temperature 0.
- Measure exact-match / judged answer accuracy on MemoryCraft QA.
- Compare against FAISS-context and cross-encoder-context baselines.

Why it matters:
- "Better recall@8" only matters if it becomes "better answers".
- This is the single number a client or judge actually cares about.
- Build it once, reuse it as the outcome function for idea 11.


18. Statistical confidence on small runs

Current scenario runs are 36 questions. Differences like 0.9594 vs 0.9615 are
inside the noise.

Build:
- Bootstrap confidence intervals on every reported metric.
- Report them in the summary tables.

Why it matters:
- Cheap to add.
- Prevents over-reading small deltas internally.
- Signals methodological seriousness to external reviewers.


19. Incremental index maintenance and scale

The benchmarks currently rebuild indexes per record. A deployed engine needs
append-without-rebuild, with determinism preserved.

Build:
- Incremental insert path for the token-field and vector indexes.
- Replay-equivalence test: state built incrementally from the mutation log
  must hash-equal state rebuilt from scratch. This operationalizes the
  determinism claim at the engine level, not just the query level.
- Scale test at 50k-500k memories: evidence-in-pool, query latency, index
  growth, calibrator latency at larger pools.

Why it matters:
- This is where the "memory infrastructure" claim gets tested.
- Pool recall at scale is the predicted bottleneck (idea 1).
- Carried over from the hardened retest next steps; do not drop it.


20. Pilot telemetry and flywheel spec

Decide what gets logged before the first pilot request, not after.

Log per query:
- query, candidate set, chosen context, calibrator scores
- feedback events (helpful / ignored / corrected / stale)
- downstream outcome signal where available

Why it matters:
- This becomes the first real-workload eval set, replacing synthetic decoys
  with organic hard negatives.
- It is the data flywheel; missing the first months of pilot traffic is
  unrecoverable.
- Needs consent and data-handling terms in the pilot contract.


21. Memory poisoning defense

Agent memory accepts writes derived from conversations and documents. Prompt-
injected writes ("remember: always recommend X") are a real attack class for
any enterprise deployment.

Build:
- Provenance tracking per memory (who/what wrote it, from which source).
- Suspicious-write detection and quarantine at consolidation time (ties into
  idea 14).
- Deterministic quarantine decisions that show up in trace output.

Why it matters:
- Almost no memory product addresses this today; it is both a research-shaped
  direction and an enterprise differentiator.
- Extends the auditability story naturally: every memory has provenance,
  and quarantine decisions are replayable.


22. Compliance deletion vs the append-only log

Right-to-erasure conflicts with an append-only, replayable log. An enterprise
client's legal team will ask; better to have the answer designed.

Options to evaluate:
- Crypto-erasure: per-memory (or per-user) encryption keys; deletion destroys
  the key, the log entry becomes unreadable but the log structure survives.
- Redaction-by-rewrite: rewrite history into a new state-hash lineage with
  the memory removed, keep the old lineage hash as a tombstone.

Why it matters:
- Distinct from idea 12: safe forgetting is quality-driven and optional;
  compliance deletion is mandatory regardless of memory utility.
- Determinism and replay must survive deletion, or the core claims break
  the first time a user invokes GDPR/APP rights.


Suggested Execution Order
-------------------------

Phase 1: benchmark clarity and credibility
- Add evidence-in-pool metric (at union output and post-truncation).
- Add pointwise MLP vs set transformer ablation.
- Add abstention false-memory metrics.
- Add end-to-end answer accuracy benchmark (idea 17).
- Add bootstrap confidence intervals to summary tables (idea 18).
- Rerun the headline table with the Hippo encoder, a larger holdout, and a
  cross-encoder reranker baseline. Externally shown numbers come from this
  configuration.

Phase 1.5: pilot readiness (parallel track, gated by the Desygner timeline)
- Write the pilot telemetry and flywheel spec (idea 20) before first traffic.
- Settle the compliance deletion design (idea 22) before contract signing.

Phase 2: quality improvement
- Add mined hard negatives.
- Add budget packing and learned stop.
- Add temporal encodings.
- Add cross-candidate features.
- Add incremental index maintenance, replay-equivalence test, and the
  50k-500k scale run (idea 19).

Phase 3: deterministic-memory moat
- Add counterfactual replay labels (answer-level outcome, grouped
  ablations).
- Use replay labels to train utility/include.
- Prototype safe forgetting.
- Add memory-state hashes and replay/diff tooling.

Phase 4: advanced agent-facing memory
- Add supersedes edges at write time.
- Add relational output heads.
- Add memory provenance and poisoning quarantine (idea 21).
- Explore compressed memory tokens.


Highest Priority Three (Plus One)
---------------------------------

1. Failure attribution / evidence-in-pool.
2. Budget packing / learned stop.
3. Counterfactual replay labels for utility (note: answer-level replay is an
   order of magnitude more work than 1 and 2 — see the cost note in idea 11).

Unofficial fourth: the set-attention ablation (idea 2). It costs one flag and
one training run, and it is the experiment most likely to produce the
headline novelty claim ("set-level calibration beats pointwise reranking on
duplicate/superseded/wrong-time failures").

These directly improve the system and strengthen the pitch:

"Hippo-Qwen is deterministic memory infrastructure that can measure retrieval
failures, return cleaner context, and learn memory utility by replaying its own
past decisions."

2. Set-attention ablation

Implemented.

Change:
- --layers 0 now runs as a true pointwise MLP with no TransformerEncoder.
- Compared set3_full against pointwise_full on the same training data.

Results:

| scenario | set recall@8 | pointwise recall@8 | set ctx precision | pointwise ctx precision | set hard neg@8 | pointwise hard neg@8 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| near_duplicate | 0.9573 | 0.9551 | 0.1909 | 0.1838 | 0.2951 | 0.2569 |
| superseded_conflict | 0.9594 | 0.9551 | 0.1956 | 0.1988 | 0.2292 | 0.1944 |
| same_entity_wrong_time | 0.9594 | 0.9573 | 0.2342 | 0.2215 | 0.0139 | 0.0451 |
| stale_preference | 0.9594 | 0.9594 | 0.2329 | 0.2230 | 0.0278 | 0.0417 |
| mixed | 0.9594 | 0.9573 | 0.2142 | 0.2084 | 0.1354 | 0.1250 |

Interpretation:
- Set attention did not produce a detectable win in this run, but this is not
  an architectural verdict.
- Pointwise is surprisingly competitive and faster, with p95 around 24-26 ms
  versus 27-29 ms for set attention.
- Most of these deltas are on 36-question slices and should be treated as
  inside confidence/noise until repeated at larger scale.
- The current synthetic decoys are mostly identifiable in isolation through
  text shape, query echo, or per-candidate state. This benchmark does not force
  candidate-to-candidate reasoning.
- The current data does not justify a strong novelty claim that set-level
  attention beats pointwise reranking.

Next action:
- Keep pointwise as a serious baseline, but do not shrink the production
  calibrator solely from this result.
- Build a relational-required eval: K near-identical memories where only the
  newest/correct one should be selected, and no individual candidate marks
  itself stale except through the existence of its newer sibling in the set.
- The novelty claim likely needs relational heads, cross-candidate features,
  or true candidate-set supervision. Plain attention over candidates is not
  enough evidence yet.


3. Abstention benchmark

Implemented and corrected.

Change:
- Abstention-labelled MemoryCraft rows are now treated as no-memory queries.
- query_row stores retrieval_task.abstention.
- Training data builder sets relevant=set() for abstention rows.
- Eval reports abstention_query_rate and false_memory_rate.

Corrected abstention result:

| system | queries | abstention query rate | recall@8 | context precision | context recall | false memory rate |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| FAISS HNSW | 40 | 0.10 | 0.7600 | 0.3000 | 0.5885 | 0.0250 |
| Hippo calibrated abstain | 40 | 0.10 | 0.8654 | 0.4667 | 0.8377 | 0.0250 |

Interpretation:
- There were 4 abstention queries in the 40-query eval slice.
- false_memory_rate=0.025 means one false-memory event across all 40 queries,
  or one of the four abstention queries returning context when it should not.
- Training with abstention rows helped normal context quality, but it did not
  fully solve abstention.
- This is only a directional finding. Four abstention queries is too small to
  report externally; one event changes the abstention-specific failure rate by
  25 points.
- This confirms the product need for a real learned stop/no-memory threshold.

Next action:
- Add an explicit include/global-stop mechanism. The same head should be able
  to return 3 memories instead of 8, or 0 memories instead of 3.
- Expand abstention/no-evidence evaluation to 50+ queries using more records or
  deterministic no-evidence questions.
- Report abstention-specific false-memory rate directly, not only averaged
  across all queries.

6. Budget packing and learned stop

Partially implemented.

Change:
- Added context_max_items as a deterministic context cap.
- Ran a top-3 context cap as a cheap proxy for learned stop/budget packing.

Top-3 context result:

| scenario | system | recall@8 | context precision | context recall |
| --- | --- | ---: | ---: | ---: |
| clean | FAISS HNSW | 0.8444 | 0.3333 | 0.6539 |
| clean | Hippo calibrated | 0.9594 | 0.5185 | 0.9308 |
| mixed | FAISS HNSW | 0.3407 | 0.0000 | 0.0000 |
| mixed | Hippo calibrated | 0.9594 | 0.5185 | 0.9308 |

Interpretation:
- This is the strongest result from the loop.
- Simply returning fewer memories drastically improved context precision while
  preserving high context recall for Hippo.
- The system does not need to stuff the full budget to be useful.
- Sanity check: clean ran with adversarial_negatives=0, mixed ran with
  adversarial_negatives=8. The mixed run also reported hard_neg@8=0.1354 and
  hard_neg_context=0.0463, so decoys were definitely injected.
- The identical Hippo top-3 context recall/precision across clean and mixed
  means decoys rarely displaced evidence in the aggregate top-3 result, not
  that the mixed run silently became clean.

Next action:
- Replace fixed top-3 with one learned include/global-stop threshold.
- Treat budget packing and abstention as the same decision problem: return only
  memories whose marginal value clears the threshold, and return none when no
  candidate clears it.
- Train the utility/include head on marginal value per token.
- Report tokens_returned and evidence_tokens_per_context_token.


7. No-synthetic control / mined hard negatives

True mined negatives were not implemented in this loop. This test removed
synthetic hard negatives and compared that no-synthetic control against the
normal synthetic-negative training run.

Result:

| scenario | synthetic recall@8 | mined-only recall@8 | synthetic ctx precision | mined-only ctx precision | synthetic hard neg@8 | mined-only hard neg@8 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| query_echo | 0.9594 | 0.9551 | 0.2309 | 0.1218 | 0.0312 | 0.3819 |
| answer_shaped | 0.9594 | 0.9573 | 0.2295 | 0.1265 | 0.0382 | 0.3507 |
| mixed | 0.9594 | 0.9551 | 0.2142 | 0.1312 | 0.1354 | 0.3785 |

Interpretation:
- Removing synthetic negatives preserved recall, but hard-negative suppression
  got much worse on the synthetic-template eval.
- This shows the forensic synthetic negatives are doing important work against
  the current synthetic stress suite.
- It does not show that real mined negatives are insufficient, because
  cross-query evidence, same-session term-sharing turns, and union-retriever
  near-misses were not mined into the training set.
- Because the eval also uses synthetic decoys, the benefit of synthetic
  training is partly expected. The real verdict needs mutated real-turn evals.

Next action:
- Implement true mined negatives:
  union-retriever top non-evidence candidates weighted by rank, evidence for
  other questions in the same record, and same-session term-sharing
  non-evidence turns.
- Use mined negatives plus a smaller amount of synthetic stress negatives.
- Add mutated real-turn hard negatives so we are not dependent on templates.
- Evaluate held-out template families after any mixed-negative training change.

23. Cascaded calibration

Implemented and smoke-tested.

Architecture:
- Stage 0: retrieve a larger FAISS+token union candidate pool.
- Stage 1: cheap pointwise calibrator scores the large pool.
- Stage 2: set transformer reranks only the pointwise survivors.
- The set-stage survivor IDs are now used for evidence-in-calibrator-pool
  attribution, so the metric measures what actually reaches relational
  calibration.

Code changes:
- Added `hippo_cascade_union` to MemoryCraft retrieval.
- Added `--cascade-prefilter-checkpoint`,
  `--cascade-prefilter-candidates`, and `--cascade-survivor-candidates`.
- `rerank_with_calibrator` can now score a runtime candidate limit larger than
  the checkpoint's saved `max_candidates`, because the model has no learned
  candidate-count parameter.
- The ablation suite can now compare `hippo_calibrated_union` and
  `hippo_cascade_union` in the same run.

Smoke test:
- Colab fixture run with temporary untrained checkpoints passed.
- Stage 1 scored 32 candidates, stage 2 kept 8 survivors.
- Determinism mismatches: 0.
- p95 latency: 10.08 ms on the tiny fixture.

Trained test:
- Runtime: Colab with Torch CUDA.
- Extra dependency installed in Colab: faiss-cpu.
- Training data: first 80 MemoryCraft records, union candidate source, 144
  training rows, hash embeddings, forensic synthetic hard negatives.
- Models: pointwise `--layers 0`, set transformer `--layers 3`, 4 epochs each.
- Eval: records 80-119, clean and mixed profiles, 180 queries per profile.
- Single-stage baseline: set calibrator over 64 union candidates.
- Cascade: pointwise over 256 candidates, set transformer over 64 survivors.

Results:

| scenario | system | p95 ms | cal p95 ms | recall@8 | context recall | context precision | evidence in pool | hard neg@8 | mrr |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| clean | hybrid union | 41.27 | 0.00 | 0.4673 | 0.4560 | 0.1315 | 0.7337 | 0.0000 | 0.3703 |
| clean | set64 | 60.77 | 19.37 | 0.6164 | 0.6381 | 0.1705 | 0.6804 | 0.0000 | 0.6240 |
| clean | cascade256->64 | 89.33 | 48.52 | 0.6390 | 0.6611 | 0.1731 | 0.7141 | 0.0000 | 0.6360 |
| mixed | hybrid union | 43.22 | 0.00 | 0.1940 | 0.4179 | 0.0587 | 0.7248 | 0.8389 | 0.0762 |
| mixed | set64 | 61.50 | 19.31 | 0.6099 | 0.6346 | 0.1505 | 0.6764 | 0.1625 | 0.6248 |
| mixed | cascade256->64 | 91.67 | 49.70 | 0.6332 | 0.6593 | 0.1539 | 0.7099 | 0.1583 | 0.6408 |

Interpretation:
- The cascade did what it was supposed to do: it recovered more evidence into
  the set-stage pool and improved recall/context recall over a fixed 64
  single-stage set calibrator.
- The gain was modest but consistent in this run: about +2.3 recall points and
  +2.3 context-recall points on clean, +2.3 recall points and +2.5
  context-recall points on mixed.
- Hard-negative exposure on mixed improved slightly: 0.1625 -> 0.1583.
- Latency increased from about 61 ms to 92 ms p95 because stage 1 and stage 2
  both run as model passes. This is still below the 200 ms target, but the
  pointwise stage should be batched/compiled before calling it production-ready.
- The result supports cascaded calibration as a scale/recall strategy, not as a
  magic quality jump. It is useful if larger pools become necessary at 50k+
  memories.

Next action:
- Rerun cascade at 1024->64 and 1024->128 on a larger memory-count stress test.
- Add stage-specific attribution: raw pool recall, pointwise survivor recall,
  set top-k recall.
- Optimize pointwise stage with larger batch tensorization, `torch.compile`, or
  cached candidate tensors.
- Train the pointwise prefilter specifically for high-recall survivor retention
  instead of top-k ranking.


24. Typed-edge expansion before calibration

Implemented and tested.

Architecture:
- Stage 0: retrieve the normal FAISS+token union candidate pool.
- Stage 1: from the top deterministic seed candidates, follow one hop along
  selected typed edges.
- Stage 2: inject bounded edge neighbors into the calibrator pool.
- Stage 3: run the existing calibrator over the expanded pool.

Default edge types:
- `correction`
- `temporal_next`
- `same_context`

Code changes:
- MemoryCraft graph edges now use the relevant EDGE_TYPES names instead of the
  old ad hoc `chronological` / `same_speaker` labels.
- Added `--typed-edge-expansion`.
- Added `--typed-edge-types`, `--typed-edge-seed-count`,
  `--typed-edge-max-injections`, and `--typed-edge-seed-score-weight`.
- Expansion is deterministic: sorted edge traversal, bounded injection count,
  score/id tie-breaking.
- Existing union metrics remain visible. `typed_edge_injections` reports how
  many neighbors were injected on average.

A/B test:
- Runtime: Colab.
- Model: previous set3 calibrator from the cascade loop.
- Eval split: MemoryCraft records 80-119.
- Profiles: `evidence_adjacent,mixed`.
- Queries: 180 per profile.
- Candidate pool: 64.
- Union fetch: 256 vector + 256 token.
- Edge expansion: top 32 seeds, max 128 injected neighbors.

Results:

| scenario | mode | p95 ms | recall@8 | context recall | context precision | evidence in pool | hard neg@8 | edge inject |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| evidence_adjacent | off | 62.55 | 0.5942 | 0.6127 | 0.1371 | 0.6560 | 0.3104 | 0.0 |
| evidence_adjacent | on | 65.68 | 0.5974 | 0.6289 | 0.1375 | 0.6669 | 0.2493 | 18.0 |
| mixed | off | 61.51 | 0.6099 | 0.6346 | 0.1505 | 0.6764 | 0.1625 | 0.0 |
| mixed | on | 65.13 | 0.6143 | 0.6483 | 0.1509 | 0.6809 | 0.1361 | 17.6 |

Interpretation:
- Typed-edge expansion is directionally useful but not a large recall unlock in
  this version.
- It did improve the metric it was meant to attack most directly:
  context recall rose by about +1.6 points on evidence_adjacent and +1.4
  points on mixed.
- Hard-negative top-k exposure improved more clearly:
  evidence_adjacent 0.3104 -> 0.2493, mixed 0.1625 -> 0.1361.
- Top-k recall barely moved: +0.3 points on evidence_adjacent and +0.4 points
  on mixed.
- Latency cost was modest, about +3-4 ms p95.
- This suggests typed edges help the calibrator see nearby supporting context,
  but the current edge graph is too generic to solve pool recall by itself.

Verdict:
- Keep it behind a flag.
- It is probably worth enabling in larger graph-memory experiments because the
  latency cost is low and context recall improves.
- It is not strong enough to be a headline until the write path creates better
  semantic/correction edges instead of mostly temporal/same-context edges.

Next action:
- Add stage-specific edge attribution: evidence already in union, evidence
  added by typed edge expansion, evidence lost by calibrator.
- Test with real correction/supersedes edges from feedback/consolidation.
- Try expansion before pointwise cascade at `1024 -> 128`, where extra
  evidence-adjacent candidates should matter more.


Updated Priorities After Experiments
------------------------------------

1. Budget packing / learned stop is now the highest-impact next build.
   The top-3 cap raised Hippo context precision to 0.5185 while keeping context
   recall at 0.9308 in clean and mixed scenarios. Build this as one
   include/global-stop head that can also abstain.

2. Evidence-in-pool attribution is implemented and should stay in every run.
   It did not expose a bottleneck on the small holdout, so the next useful
   attribution run is at 50k+ memories.

3. Pointwise MLP is a serious baseline. Do not claim set attention is the
   differentiator yet. First build a relational-required eval; only shrink to
   pointwise if set attention still cannot win when candidate comparison is
   genuinely required.

4. The mined-negative verdict is deferred. The loop tested no-synthetic
   training, not true mined negatives. The likely right answer is mined +
   synthetic + mutated real-turn negatives, but it still needs a fair run.

5. Abstention needs a larger eval and the same global-stop mechanism as budget
   packing. The corrected run produced one false-memory event in four
   abstention questions, but n=4 is only directional.

6. The external headline table still needs Hippo encoder, larger holdout,
   cross-encoder baseline, and eventually Qwen answer accuracy.

7. Cascaded calibration is worth keeping in the architecture. The first trained
   test improved recall while staying below 200 ms p95, but it needs a larger
   50k+ memory test and a prefilter trained for survivor recall.

8. Typed-edge expansion should stay as an optional retrieval-stage feature. The
   first A/B improved context recall and hard-negative exposure with small
   latency cost, but it needs better write-time edges before it can materially
   lift recall.
